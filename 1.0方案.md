# 今晚吃什么 · 1.0 方案

锁定 1.0 的范围、核心闭环、输入/输出契约与技术形态。需求全集见 [[需求-今晚吃什么]]；衍生功能列为 fast-follow，不在 1.0。未决项见 [[待澄清问题]]。

## 1.0 范围（已锁定）

- **引擎**：LLM 为主——晚餐答案、营养理由、口吻全部由模型 API 生成，不自建营养数据库
- **平台**：网页优先（小程序作为 fast-follow）
- **功能**：核心闭环 only

**边界外（fast-follow，不在 1.0）**：拍照识别、放纵博弈余量研判、二次传播趣味输出（阴阳暗讽 / 开盲盒 / 百大UP吃饭寄语）、小程序端。

## 核心闭环

用户输入早/午餐（一句话自然语言）→ 模型综合 [用户画像 + 今日已吃 + 当下口味/心情] → 输出决断式晚餐答案。

1. **输入**：用户画像（首次采集，越少越好）+ 今日早/午餐一句话 + 可选当下需求（"嘴里没味" / "心情不好"）
2. **处理**：单次 LLM 调用，结构化 prompt
3. **输出**：固定结构（见下）
4. **交互**：不想吃 → 换一个；想吃 → 去哪买/怎么做

## 输出契约

- **第一行**：「今晚吃 XXXX。」—— 一个明确答案，决断，不给选项清单
- **第二行**：营养角度原因 —— 精炼，一句话
- **操作**：➡️不想吃 → 换一个；➡️想吃 → 去哪买 / 怎么做

## 输入设计（最小信息原则）

- **首次**：身高、体重、性别、年龄 → 存为用户画像
- **每次**：早/午餐一句话；可选当下口味/心情
- 在「推荐准确」与「输入轻」之间取平衡：除画像四项外不强制其它输入；早/中餐用一句话自然语言即可

## 推荐引擎（LLM 为主）

- 单次模型调用生成「答案 + 营养理由」
- **prompt 组成**：系统角色（决断式营养顾问）+ 用户画像 + 今日已吃 + 当下需求 + 输出格式约束
- 营养均衡判断交给模型，不自建食物营养库
- **已知风险**：模型不够决断 / 给选项 / 医疗化措辞 / 营养判断不准 → 用 prompt 强约束兜底，细节见 [[待澄清问题]]

## 技术形态（网页优先）

- **前端**：网页 UI，单页轻交互（输入框 + 结果卡片 + 「换一个」）
- **后端最小验证**：模型 API 接入 + 用户画像存储 + 运行时存储（会话/跨次记忆）
- **前后端映射**：用户输入 → 后端拼 prompt → 调模型 → 解析为输出契约 → 前端渲染卡片

## 里程碑

- **M1 闭环打通**：写死画像，单轮「输入早/午餐 → 输出晚餐答案 + 理由」，验证决断式输出与营养理由质量
- **M2 画像 + 记忆**：首次采集画像并存储；跨次记忆（如「这周没吃某品类，是不是该吃一顿」）
- **M3 交互补全**：「换一个」「怎么做 / 去哪买」
- **M4 体验打磨**：决断度、理由精炼度、措辞口吻调优
